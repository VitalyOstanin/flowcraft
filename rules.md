# Правила работы с проектом FlowCraft

## ВАЖНО: Обязательность соблюдения правил

ВСЕ правила, описанные в данном файле rules.md, являются ОБЯЗАТЕЛЬНЫМИ для соблюдения при работе с проектом FlowCraft. Нарушение любого из правил недопустимо.

## 1. Общение с пользователем

Общаться с пользователем всегда на русском языке. Использовать обращение на вы с маленькой буквы без кавычек.

## 2. Создание документации

При создании md файлов:
- Добавлять TOC (содержание)
- Писать содержимое на русском языке
- Не использовать эмодзи

## 3. Работа с Git

Коммиты без разрешения делать НЕЛЬЗЯ!

## 4. Управление зависимостями

В проекте предпочитать uv вместо pip для управления зависимостями:
- `uv pip install package` вместо `pip install package`
- `uv pip install -r requirements.txt` для установки зависимостей
- `uv run python script.py` вместо `python script.py` для запуска скриптов
- `uv run python -c "код"` вместо `python -c "код"` для отладки и тестирования

## 5. Управление задачами

Выполненные задачи помечать в файле плана как выполненные.

## 6. Подход к решению задач

Все задачи доводить до конца. Не торопиться, выбирать правильное решение, а не быстрое.

## 7. Управление сложными задачами

Совсем сложные для немедленной реализации вещи записывать в файл TODO.md

## 8. Использование эмодзи

Не использовать эмодзи в проекте, за исключением зеленой галочки и красного креста в тестах и проверках.

## 9. Согласование с пользователем

При согласовании с пользователем разделить проблемы на отдельные пункты и проходить согласование отдельно по каждому пункту.

## 10. Детализация формулировок

Давать подробные формулировки вместо общих фраз. Вместо "слишком агрессивное завершение" описывать конкретно "слишком агрессивное завершение: происходит то-то конкретное, проявляется в виде того-то, приводит к таким-то последствиям".

## 11. Обновление документации

После изменений в проекте обязательно обновлять:
- README.md - если изменилась функциональность или структура
- TODO.md - если добавились или выполнились сложные задачи  
- Changelog.md - для фиксации всех изменений

## 12. Организация тестов

Все тесты должны находиться в подкаталоге tests/. Тесты в корне проекта недопустимы.

## 13. Перечитывание правил

После каждого запроса в LLM ОБЯЗАТЕЛЬНО перечитывать rules.md и не забывать правила оттуда, повторять их снова. Агенту нельзя забывать ни одно правило, включая "не делать коммит без разрешения" и все остальные правила из данного файла.
